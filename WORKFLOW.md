# WORKFLOW.md

Документ описывает процесс совместной работы пользователя, ChatGPT и Codex в проекте Bitrix Sales.

## Роли

| Роль | Ответственность |
|---|---|
| Пользователь | Владелец продукта и тестировщик |
| ChatGPT | Архитектор, планировщик и приемщик |
| Codex | Исполнитель |

ChatGPT принимает продуктовые и архитектурные решения, готовит задачи и проверяет результат.

Codex реализует поставленную задачу, запускает проверки, обновляет документацию и пишет отчет.

## Основные правила

- Сначала анализировать и проектировать, потом ставить задачу.
- Не придумывать факты.
- Если что-то не проверено, писать об этом явно.
- Разделять факты, предположения и неизвестное.
- Не добавлять задачи без запроса пользователя.
- Не расширять объем работ самостоятельно.
- Общение с пользователем вести на русском, если не указано иное.
- Промпты для Codex писать только по прямому запросу пользователя и только на английском.
- Задачи должны быть достаточно крупными, чтобы закрывать проверяемую продуктовую или техническую веху. Не дробить работу на микросогласования, если нет реального риска.

## Источники истины

Приоритет источников:

1. Последнее сообщение пользователя.
2. Текущие файлы проекта в GitHub.
3. `.ai/task.md`.
4. `.ai/report.md`.
5. Git-история и diff.
6. `WORKFLOW.md`.
7. `AGENTS.md`.
8. `SPEC.md`.
9. Документация проекта в `docs/` и README-файлах.

Память и догадки не считаются фактами.

Если актуальных файлов нет или доступ к ним отсутствует, ChatGPT должен запросить их у пользователя.

## Режимы работы

### Обсуждение

Используется для анализа идеи, требований и вариантов решения.

ChatGPT:

- уточняет цель;
- определяет ограничения;
- объясняет компромиссы;
- не создает задачу без запроса пользователя;
- не пишет промпт для Codex без запроса пользователя.

### Проектирование

ChatGPT предоставляет:

- подход к решению;
- влияние на архитектуру;
- затронутые модули;
- риски;
- стратегию проверки;
- открытые вопросы.

Если не хватает фактов, влияющих на продукт, данные, безопасность или объем работ, нужно остановиться и спросить пользователя.

### Подготовка задачи

ChatGPT формирует `.ai/task.md` и делает planner-коммит.

Задача должна содержать:

- цель;
- факты;
- предположения;
- неизвестное;
- объем работ;
- что не входит в задачу;
- ограничения;
- критерии приемки;
- проверки;
- hard gate перед commit.

### Выполнение задачи

Codex выполняет только задачу из `.ai/task.md`.

Codex не должен:

- расширять scope;
- выполнять соседние задачи;
- менять `.ai/task.md` без явного запроса;
- использовать `git add .` без явного разрешения;
- трогать неизвестные локальные изменения;
- коммитить секреты, локальные базы, raw data, generated data или build artifacts.

### Проверка результата

Если пользователь пишет `Проверяй`, ChatGPT проверяет результат через GitHub:

- сравнивает `main` с последним planner-коммитом;
- читает `.ai/report.md`;
- смотрит измененные файлы и ключевые участки кода;
- проверяет выполнение acceptance criteria;
- проверяет выполненные проверки;
- ищет нарушения архитектурных, продуктовых и security-границ.

Review должен учитывать не только кодовые баги, но и продуктовые несостыковки: вводящие в заблуждение метрики, непонятные пользовательские состояния, отсутствие loading/error states, неверный operator flow.

Если результат принят, ChatGPT говорит об этом в чате. Отдельный `accept:` commit по умолчанию не нужен.

## Файлы обмена

```text
.ai/
  task.md
  report.md
```

`.ai/task.md` — входной контракт для Codex.

`.ai/report.md` — отчет Codex о выполнении.

## Формат идентификатора задачи

```text
TASK-YYYY-MM-DD-01
```

## Формат задачи

```md
# Task: TASK-YYYY-MM-DD-01

Status: planned
Created from: <commit/current main/unknown>

## Title

## Goal

## Facts

## Assumptions

## Unknowns

## Scope

## Out Of Scope

## Constraints

## Acceptance Criteria

## Checks

## Hard Workflow Gate
```

## Формат отчета

```md
# Отчет: TASK-YYYY-MM-DD-01

Статус: done | partial | blocked | failed

## Кратко

## Измененные файлы

## Запущенные проверки

## Факты

## Предположения

## Неизвестное

## Риски или следующий шаг
```

## Статусы

| Статус | Значение |
|---|---|
| planned | Задача поставлена |
| done | Выполнено |
| partial | Выполнено частично |
| blocked | Требуется пользователь, доступ или внешнее состояние |
| failed | Результат непригоден |

Acceptance фиксируется сообщением пользователя/ChatGPT в чате, а не обязательным commit.

## Протокол коммитов

```text
planner: TASK-... <task>
codex: TASK-... <task>
```

`planner:` — ChatGPT поставил задачу.

`codex:` — Codex выполнил задачу.

`accept:` commit не используется по умолчанию. Его можно делать только если пользователь явно попросит фиксировать приемку коммитами.

История должна оставаться линейной.

## Правила старта Codex

Перед началом:

```bash
git log --oneline -5
git status --short
```

Работа разрешена только если последний релевантный коммит:

```text
planner: <task-id> <task>
```

Если последний релевантный коммит уже `codex: ...`, задачу повторно не выполнять.

Неизвестные локальные изменения не трогать.

## Правила завершения Codex

После выполнения:

1. Обновить `.ai/report.md`.
2. Запустить необходимые проверки.
3. Проверить изменения:

```bash
git status --short --branch
git diff --stat HEAD
git diff --name-only --cached
git diff --check -- ':!AGENTS.md' ':!.ai/task.md'
```

4. Добавить только файлы текущей задачи и `.ai/report.md`.
5. Убедиться, что не staged `.env`, базы, raw/generated data, build artifacts, caches, logs, `node_modules`, `frontend/dist`, `ui-kits/`.
6. Сделать коммит:

```bash
git commit -m "codex: <task-id> <task>"
```

## Размер задач

По умолчанию задачи должны быть крупными проверяемыми вехами.

Хорошая задача закрывает один цельный результат, например:

- backend/data readiness;
- один пользовательский frontend screen;
- operator flow;
- один отчет;
- инфраструктурный запуск проекта;
- документационное обновление workflow.

Не нужно дробить задачу на микрошаги, если:

- нет отдельного продуктового решения;
- нет высокого риска данных или безопасности;
- нет необходимости ручной проверки между шагами.

Остановиться и спросить пользователя нужно, если требуется бизнес-решение, меняется аналитическая логика, есть риск записи в Bitrix, затрагиваются секреты/данные, или текущий scope становится неоправданно широким.

## Operator Flow Checks

Если задача влияет на запуск, UI или пользовательский сценарий, проверять нужно не только unit/build tests.

Нужно проверить применимый operator flow, например:

```bash
docker compose config
docker compose up --build -d
curl -f http://localhost:8000/health
curl -f http://localhost:5173/
docker compose down -v
```

Для UI-задач дополнительно проверять:

- понятное состояние без данных;
- loading state;
- error state;
- empty state;
- пользовательский action;
- обновление данных после action;
- отсутствие вводящих в заблуждение метрик.

Если Docker, сеть или live-доступ недоступны, это должно быть явно написано в `.ai/report.md`.

## Локальные данные и секреты

`.env`, DuckDB, Parquet, CSV, raw Bitrix data, generated data и snapshots не коммитятся.

Наличие `.env` означает только, что backend может подключиться к Bitrix. Наличие `.env` не означает, что локальная база уже подготовлена.

Docker Compose только запускает сервисы. Он не должен автоматически вызывать Bitrix или обновлять данные без явного действия пользователя.

Если локальная база не подготовлена, UI должен показывать понятное состояние и пользовательское действие, например `Обновить из Bitrix`.

## Bitrix Safety

Bitrix используется только read-only.

Запрещены CRM write methods, включая:

```text
crm.*.add
crm.*.update
crm.*.delete
crm.*.set
```

Любая задача, связанная с Bitrix, должна явно фиксировать:

- какие методы разрешены;
- какие методы запрещены;
- запускался ли live Bitrix вызов;
- что секреты и row-level персональные данные не выводились.

## Правила промптов

ChatGPT не выдает промпт для Codex до прямого запроса пользователя.

Допустимые запросы:

```text
Дай prompt для Codex.
Сформируй задачу для Codex.
Продолжай через workflow.
```

Промпт для Codex всегда писать на английском.

## Команды пользователя

Для ChatGPT:

```text
Проверяй
Давай задачу
Готовь задачу
Что делаем дальше?
```

Для Codex:

```text
Выполни задачу из .ai/task.md согласно workflow.
```

## Формат ответов ChatGPT

Для простых вопросов отвечать кратко.

Для review:

1. Блокеры, если есть.
2. Что проверено.
3. Вердикт: принимать или нет.
4. Следующий шаг, если он очевиден.

Не перегружать пользователя деталями, если они уже есть в `.ai/report.md`.

## Условия остановки

ChatGPT должен остановиться и запросить уточнение, если:

- продуктовая логика неоднозначна;
- отсутствуют необходимые файлы;
- есть риск архитектурного конфликта;
- требуется широкий рефакторинг;
- задача добавляет непрошенную функциональность;
- не хватает фактов для принятия решения;
- есть риск записи в Bitrix или утечки данных.

Codex должен поставить `blocked`, если:

- отсутствует `.ai/task.md`;
- последний релевантный коммит не `planner:`;
- есть неизвестные локальные изменения, мешающие работе;
- не хватает файлов или доступа;
- критерии приемки невозможно проверить.

## Хороший цикл

- одна крупная проверяемая веха;
- явные факты, предположения и неизвестное;
- четкий out of scope;
- проверяемые acceptance criteria;
- релевантные automated checks;
- operator-flow checks для UI/runtime задач;
- небольшой и связный diff;
- отсутствие несвязанного рефакторинга;
- актуальная документация;
- понятное решение о приемке в чате.
